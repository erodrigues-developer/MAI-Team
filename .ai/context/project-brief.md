# Project Brief

This file is the product source of truth for all agents (`pm`, `architect`, `dev`, `qa`).

It should be:
- initialized during architect bootstrap
- updated only when product direction meaningfully changes
- kept short, objective, and decision-oriented

## Purpose
Describe the core purpose of this project.

What problem does this system solve?  
Who benefits from it?  
Why does it exist?

## Target Users
Who are the primary users of the system?

Examples:
- developers
- product teams
- internal tools users
- customers of a SaaS platform

Describe their main needs.

## Core Problem
Describe the main problem the system is solving.

What pain exists today?  
What inefficiency or limitation does this project address?

## Value Proposition
Why is this solution valuable?

What does it improve?

Examples:
- faster development cycles
- reduced operational complexity
- improved automation
- improved developer experience

## Product Scope
Define what the system is responsible for.

## Out of Scope
Define what the system is not responsible for.

This section is mandatory to avoid uncontrolled scope expansion.

## Key Features
List the main capabilities expected from the system.

## High Level Workflow
Describe the expected product workflow.

## Product Principles
Define the product philosophy.

Examples:
- keep automation transparent
- maintain human oversight
- prefer simplicity over complexity
- integrate with existing developer tools
- prioritize reliability

## Success Metrics
How will we know the system is successful?

Examples:
- reduced time from story to PR
- increased automation coverage
- fewer manual coordination steps
- improved code review consistency

## Project Stage
Current stage of the project.

Example:
- experimental
- MVP
- beta
- production

## Future Vision
Where could this project evolve?

## Update Rule
Update this file only when there is:
- significant product direction change
- target audience change
- strategic change
- scope redefinition

Do not update this file for every story.
